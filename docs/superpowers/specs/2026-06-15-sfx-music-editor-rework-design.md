# SFX & Music Editor Rework — Design

- **Status:** Draft for review
- **Date:** 2026-06-15
- **Scope:** `rico8-console` editors (`sfx.rs`, `music.rs`), with supporting additions in
  `rico8-runtime` (`assets.rs`, `audio.rs`, `pico8.rs`).
- **Goal:** Make RICO-8's SFX and music editors look and behave like PICO-8's, which the current
  editors do not. In particular the music editor today only lets you pick an SFX number per channel
  — it shows no notes and no timing — so you cannot see or shape the music from it.

## Visual target (ground truth)

PICO-8 and RICO-8 share the same 128×128 screen and the same 16-colour palette, so we pixel-match
rather than approximate. The reference framebuffers below were recovered from the user's PICO-8
screenshots (each was an exact 14× upscaled PICO-8 screen; we cropped the screen region and
downsampled to the true 128×128, quantised to the palette). They are the literal target.

- `…-assets/pico8_music_128.png` — real PICO-8 music editor.
- `…-assets/pico8_sfx_pitch_128.png` — real PICO-8 SFX editor, pitch mode.
- `…-assets/mockup_music.png`, `mockup_sfx_pitch.png`, `mockup_sfx_tracker.png` — RICO-8
  reproductions rendered through the real framebuffer + font + palette (proof the layout is
  reproducible with our primitives). Generated by the throwaway example
  `rico8-runtime/examples/mockup_editors.rs`, which is **deleted before implementation lands**.

We have no PICO-8 screenshot of **tracker mode** or the **Wave Designer**; their exact pixel layout
must be pinned against a real PICO-8 reference during implementation (recover one the same way, or
request a screenshot). Everything else is matched to the recovered framebuffers.

## Decisions (locked with the user)

1. Rework **both** editors together, pixel-matching PICO-8's editor body. Keep RICO-8's shared red
   tab bar on the top-right (the other three editors share it); only its **active-tab highlight**
   changes (see below).
2. Music columns are a **view** of each channel's SFX notes; you author notes in the SFX editor,
   reached via the channel **pencil** ("SFX edit button"). No inline note editing in the music
   editor. (This matches PICO-8 exactly.)
3. **Include the Wave Designer**: SFX 0–7 can be drawn custom-waveform instruments. This is the one
   feature requiring a data-model + synth addition.
4. **Live playhead** in both editors, driven by the synth's real per-channel step.

## Non-goals

- No change to the cartridge format beyond the additive Wave Designer field (serde-optional, so old
  carts still load).
- The pre-existing tick-rate discrepancy (RICO-8 times steps at 1/128 s; PICO-8 is ~1/120 s, i.e.
  183 samples at 22050 Hz) is **out of scope** here. Noted for a possible separate change.
- PICO-8's configurable snap scales: ship the default (C-minor-pentatonic) only.

## Shared chrome

### Top bar (rows 0–7, red `col 8`)

- **Right side — editor tabs.** Five icons at x = 83, 92, 101, 110, 119 (`tab_x(i)` already used by
  `ui::draw_tab_bar`). **Fix:** the active tab is distinguished by **foreground colour only** —
  peach (`col 15`) for active, dark-purple (`col 2`) for inactive — with **no background box**. The
  current `ui::draw_tab_bar` paints a `DARK_PURPLE` box behind the active icon, which melds with the
  inactive icons' colour; remove that box. This touches the shared `ui.rs`, so all five editors gain
  the cleaner highlight (acceptable, strictly an improvement).
- **Left side — two view-mode buttons** (SFX editor and music editor both show them):
  - a small **bar-chart** icon at x ≈ 4 = **pitch mode**;
  - a **3×3 dot-grid** icon at x ≈ 14 = **tracker mode**.
  Active = peach, inactive = dark-purple. `TAB` toggles them. In the SFX editor they switch the
  note view between pitch and tracker. In the music editor they switch how each channel column
  renders its notes (tracker is the default; supporting pitch-style columns is optional and may be
  deferred — flag).

### Bottom bar (rows 121–127, red)

A thin red status strip with a one-line hint, as today (`ui::status_bar`).

## Music editor (`editor/music.rs`)

Reference: `pico8_music_128.png`. Layout, top to bottom.

### Pattern strip (rows ~9–21)

- `"pattern"` label at x = 4, y = 12 (`col 6` light grey).
- **Pink ◀** at x ≈ 36 (apex left) and **pink ▶** at x ≈ 94 (apex right) scroll the pattern
  navigator through patterns #0–63.
- **Five pattern boxes** at x = 40, 51, 62, 73, 84 (9 px wide, rows 12–20), each a 2-digit pattern
  number on a **black-filled button**. The current pattern gets a white frame; others use light-grey
  digits. (PICO-8 draws a 1px light-grey *border* around each box, but rico8's fatter font merges
  with a border at this size, so we drop the inner border and rely on the black fill for contrast —
  a deliberate readability divergence, confirmed with the user.)
- **Activity dots** above each box (y = 10): up to four 1-px dots, one per channel, drawn **only for
  channels that are enabled** in that pattern (PICO-8 omits inactive ones — do not draw grey
  placeholders). Channel colours: **ch0 orange (9), ch1 yellow (10), ch2 green (11), ch3 blue (12)**.
- **Flow-control buttons** to the right of ▶, at x ≈ 100 / 108 / 117:
  - **loop-start** — blue (`col 12`) right-arrow-into-a-bar. Marks this pattern as a loop target.
  - **loop-back** — dark-blue (`col 1`) bar-then-left-arrow. At this pattern's end, jump back to the
    nearest earlier loop-start (or pattern 0 if none).
  - **stop** — dark-blue filled square. End the song after this pattern.

Song chaining (already implemented correctly in `audio.rs::advance_music`, keep): at a pattern's
end, advance to the next pattern unless STOP is set, LOOP-BACK is set (→ nearest loop-start / 0), or
the next pattern is empty (→ stop). `MusicPattern`'s `loop_start` / `loop_back` / `stop_at_end`
already map to these three flags.

### Channel headers (rows ~23–29)

Four columns; note-panel x-bases ≈ `[2, 33, 64, 95]`, ~30 px wide. Per channel, above its panel:

- **Enable toggle** — a 5×5 light-grey box at panel_x+2 with a centre pixel when on (PICO-8 draws
  it radio-like). Click toggles the channel on/off for this pattern (`MusicPattern.channels[ch]`
  becomes `Some(sfx)` / `None`).
- **SFX-number box** — a black box at panel_x+9 showing the 2-digit SFX index assigned to the
  channel. **Left-click = +1, right-click = −1** (wrap 0–63). Shift-click = ±4 (PICO-8 convention).
- **Pencil** ("SFX edit button") at panel_x+20, lavender (`col 13`). Clicking it switches to the
  **SFX editor** with this channel's SFX selected, so you edit its notes/speed/etc. there.
- **Disabled channels** show only an empty 5×5 box (no number, no pencil).

### Note columns (rows ~31–118)

- Enabled channel → a **black** panel showing the referenced SFX's notes as a tracker column;
  disabled channel → a **black-bordered** box on the grey bg (no fill).
- Row height: PICO-8's note rows are **not a fixed 6 px** — measured captures show ~8 px (11 rows)
  for full-length SFX and ~16 px when the SFX is short (LEN), i.e. the display **scales the SFX's
  playable length to fill the column**. Match this (verify the exact rule when implementing); the
  mockup uses 8 px for the dense case. **Empty (silent) steps render as a faint dotted line**, not
  blank.
- **Per-row note cell**, field colours matched to PICO-8:
  - note letter + accidental — white (`col 7`); `#` for sharps, `-`/blank for naturals;
  - octave digit — light grey (`col 6`);
  - instrument digit — pink (`col 14`), or **green** (`col 11`) when it is a custom instrument
    (SFX 0–7 — PICO-8 tints custom instruments green; RICO-8 currently uses yellow, switch to
    green to match);
  - volume digit — blue (`col 12`);
  - effect — a grey `.` when 0, else the digit in orange (`col 9`).
  - empty step — dim dashes.
- **Live playhead:** during playback, highlight each channel's currently-sounding step (yellow row).
  Because each channel's SFX has its own speed, the highlighted rows differ per channel — this is
  the visual cue for relative timing. Pattern length is the leftmost non-looping channel's duration
  (already implemented in `audio.rs::start_pattern`).

### Music-editor input summary

- Mouse: pink arrows scroll patterns; click a pattern box to select; click enable toggle; left/right
  (and shift) click the SFX-number box; click the pencil → SFX editor; click flow buttons.
- Keyboard: `PgUp/PgDn` change pattern; `Space` play/stop; `Tab` switch note-view mode; pattern
  range copy/paste via click → shift-click → `Ctrl-C` / `Ctrl-V` (optional, can be a follow-up).

## SFX editor (`editor/sfx.rs`)

Three sub-modes, selected by the top-left buttons / `TAB` (Wave Designer only for SFX 0–7, reached
by its own toggle): **pitch**, **tracker**, **wave designer**.

### Shared header (rows ~9–16)

- **SFX selector** — `◀ NN ▶` (pink arrows, white number) at x ≈ 4–23. Keys `-` / `+`.
- **`spd` + black box** at x ≈ 35/49 — speed 0–255 (treated as ≥1). Left-click +1 (slower),
  right-click −1, shift ±4, or drag. Keys `<` / `>`.
- **`loop` + two black boxes** at x ≈ 72/87/103 — loop start / end (0–31). Looping off when
  `start ≥ end`; when **end = 0**, the first box is a **length** (label flips to `len`), which is how
  patterns get non-32 lengths. (`Sfx.loop_start` / `loop_end` already model this; `audio.rs` already
  honours it.)
- **Far-right `∿` toggle** (x ≈ 115) — **Wave Designer** toggle, enabled only for SFX 0–7. Enters
  waveform-draw mode.

### Pitch mode

Reference: `pico8_sfx_pitch_128.png`.

- **`:pitch` label** (lavender) + **8-waveform palette** at x = 45 + 9·w (8 px boxes). The selected
  waveform has a red box; others light-grey. Each box draws its waveform glyph (triangle, tilted
  saw, saw, square, pulse, organ, noise, phaser). Click selects; **shift-click applies that waveform
  to all 32 notes**. A trailing small toggle (the "o") flips the palette between icon and numeric
  display (nice-to-have; may defer).
- **Pitch graph** (rows ~28–99): 32 vertical bars on black; **bar height = pitch (0–63)**, coloured
  by that note's instrument; a marker on each bar top. **Click-drag** paints pitch across steps with
  the selected instrument; **shift** = change pitch only (keep each note's instrument); **ctrl** =
  snap to C-minor-pentatonic; **right-click** = eyedropper (grab a note's instrument).
- **`:volume` strip** (rows ~101–118): one marker per step, vertical position = volume 0–7; drag to
  set; shift-click = apply to all. (PICO-8's far-right `:volume` icon and a pitch-mode effects row
  are **undocumented**; render the volume strip, treat the effects row as optional/deferred.)

### Tracker mode

Reference: `mockup_sfx_tracker.png` (built from docs; pin against a real PICO-8 tracker screenshot).

- Two columns of 16 steps (0–15, 16–31) on black, each row: **step index** (gutter, dim) + note cell
  (same field layout/colours as the music columns).
- Dotted gridline every 4 steps; cursor box on the focused field; loop/len range marker in the
  gutter; **live playhead** highlights the sounding step.
- **Filter switches** (PICO-8 0.2.x), accessible in tracker mode, in the gutter between columns:
  **NOIZ, BUZZ** (on/off) and **DETUNE, REVERB, DAMPEN** (levels 0/1/2). These already exist in
  `Sfx` (`noiz, buzz, detune, reverb, dampen`) and are edited today; keep, restyle to match.
- Note entry: piano keys `zsxdcvgbhnjm` (lower) / `q2w3er5t6y7ui` (upper); **shift** transposes ±1
  octave; digit keys set instrument/volume/effect on the focused field; cursor auto-advances on note
  entry; `Backspace`/`Delete` or volume 0 clears a step.

### Wave Designer mode (SFX 0–7 only) — new

Functional description from the PICO-8 docs (manual §2.5 + 0.2.6 notes); **exact pixel layout TBD,
pin against a PICO-8 reference**:

- A canvas showing the SFX's custom waveform as a drawn signal around a centre axis. Draw samples
  with the mouse.
- A **bass** toggle (pitch the waveform an octave down).
- When any note (in any SFX) references this SFX 0–7 as a custom instrument, the synth plays it using
  this drawn waveform.

## Data-model changes (`rico8-runtime/src/assets.rs`)

Add a custom-waveform field to `Sfx` (additive, serde-optional → old carts load unchanged):

```rust
pub struct Sfx {
    // … existing fields …
    /// When `Some` (only meaningful for SFX 0..8), this slot is a drawn
    /// custom-waveform instrument: signed samples in roughly -16..=15 played
    /// as the timbre by notes that reference it. `None` = a normal note SFX.
    #[serde(default)]
    pub custom_wave: Option<CustomWave>,
}

pub struct CustomWave {
    pub samples: [i8; 32],   // count/range to confirm against PICO-8's editor
    pub bass: bool,          // octave-down toggle
}
```

- Exact **sample count (32 vs 64)** and **value range** must be confirmed against PICO-8's Wave
  Designer; pick to match its drawable resolution. PICO-8 documents a "64-byte looping waveform"
  with values cropped to ~-96..96 — likely 32 samples (one per visual column) is the right RICO-8
  model; verify.
- `validate()` needs no new constraint (the field is optional and fixed-size).
- This does not change `Note`, `MusicPattern`, or any existing field.

## Runtime / synth changes (`rico8-runtime/src/audio.rs`)

1. **Expose the live playhead.** Add `Synth::channel_step() -> [Option<usize>; CHANNELS]` returning
   each live voice's current `step` (and a matching `AudioHandle` passthrough). The editors poll it
   each frame to draw the playhead. Pure read; no behaviour change.
2. **Play drawn waveforms.** In `Voice::sample`, when a note resolves to a custom instrument whose
   referenced SFX has `custom_wave: Some(w)`, synthesise from `w.samples` (linearly interpolated at
   the note frequency) instead of a built-in `tonal_wave`; apply `w.bass` as an octave-down. The
   existing custom-instrument path (borrow another SFX's built-in waveform) remains the fallback when
   `custom_wave` is `None`.
3. No change to timing, sequencing, or the existing filters.

## Import (`rico8-runtime/src/pico8.rs`)

PICO-8 marks a waveform-instrument SFX with a high bit and stores its samples in the note bytes. The
importer should detect that and populate `custom_wave` instead of `notes` for SFX 0–7. (Confirm the
exact PICO-8 encoding when implementing Wave Designer; the rest of the SFX/music import is unchanged.)

## Implementation outline

1. **`ui.rs` (console):** active-tab highlight → foreground-only; add the left-side pitch/tracker
   mode buttons; small icon helpers (arrows, radio, pencil, flow buttons, waveform glyphs).
2. **`assets.rs`:** add `CustomWave` + `Sfx.custom_wave`; tests for serde round-trip.
3. **`audio.rs`:** `channel_step()`; drawn-waveform synthesis + bass; tests.
4. **`editor/music.rs`:** full rewrite to the layout above (pattern strip, headers, view columns,
   playhead, pencil → SFX editor handoff). Needs a way to ask the shell to switch to the SFX editor
   with a given SFX selected (a small return-signal or shared selection on the shell).
5. **`editor/sfx.rs`:** restructure into pitch / tracker / wave-designer modes with `TAB`; build the
   pitch graph + volume strip and the Wave Designer canvas; keep tracker entry + filters; playhead.
6. **`pico8.rs`:** import PICO-8 waveform instruments into `custom_wave`.
7. Delete `rico8-runtime/examples/mockup_editors.rs` and the `mockups/` scratch dir.

The music ↔ SFX handoff (item 4) is the one cross-cutting bit: the pencil and the music view-mode
need the shell to change editor + selected SFX. Plumb a minimal shared "selected SFX" on the shell
(or a return value from `music_ed` consumed in `shell.rs`).

## Pixel-fidelity acceptance test (the QA gate)

Because "match PICO-8 exactly" is the requirement, fidelity is checked **mechanically**, not by eye:

1. Recover PICO-8's true 128×128 framebuffer for each editor view from a screenshot (crop the
   screen region, downsample by the integer scale, quantise to the palette) — the procedure used to
   produce `…-assets/pico8_*_128.png`.
2. Render the corresponding RICO-8 editor view to a framebuffer and **pixel-diff** it against the
   PICO-8 reference, ignoring (a) text whose *content* is example data and (b) font-glyph bitmap
   differences (RICO-8's font ≠ PICO-8's). Everything else — panel fills, element positions, chrome
   colours, icon pixels — must match.
3. Chrome that must be pixel-exact (flow buttons, mode icons, arrows) is **lifted verbatim** from
   PICO-8's framebuffer as pixel grids and blitted, rather than hand-drawn.

This diff harness is the regression test for the editors. The throwaway
`examples/mockup_editors.rs` proved the approach (music chrome diff converged from 3938 → ~570
non-font px; SFX from 11159 → mostly example-data/glyph px). Capture PICO-8 references for the
tracker mode and Wave Designer too, so all editor views are gated the same way.

## Testing

- `assets.rs`: `custom_wave` serde round-trip; old cart (no field) still deserialises.
- `audio.rs`: a drawn-waveform instrument is audible; `channel_step()` advances and clears with the
  voice; bass drops an octave.
- Editors are UI; cover logic-bearing helpers (pattern/flow toggling, SFX-number wrap, loop/len
  edits, note-cell formatting) with unit tests as today. Visual fidelity is verified by rendering
  against the reference framebuffers.

## Open items / flags (from research; verify when implementing)

- **Wave Designer exact layout + sample count/range** — no PICO-8 reference screenshot yet; pin it.
- **Pitch-mode effects row** and the `:volume` row's far-right icon — undocumented in PICO-8; defer
  or confirm empirically.
- **Pencil → SFX editor** semantics — confirmed to exist (PICO-8 changelog "SFX edit buttons on
  pattern channels"); the precise jump behaviour is inferred, matches PICO-8's UI.
- **Music-editor pitch-style columns** — PICO-8 lets `TAB` change the column view in the music
  editor; tracker view is the default and sufficient for v1, pitch-style columns optional.
- **Custom-instrument tint** — switch RICO-8's custom-instrument digit from yellow to **green** to
  match PICO-8.
